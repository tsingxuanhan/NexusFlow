## 一、实验目标

### 1.1 核心问题

**"全球10大经济体的宏观经济韧性评估与风险预警"**

这是一个典型的**高认知负荷、多维度、跨领域**的复杂分析任务：
- 需要同时处理15个宏观经济指标 × 10个国家 × 51年时间序列
- 涉及经济增长、价格稳定、财政健康、外部平衡4大分析维度
- 需要领域知识解读（制裁、房地产危机、人口结构等特殊因素）
- 需要交叉验证（不同指标间的一致性检验）
- 需要生成可视化与结构化报告

### 1.2 为什么这个任务适合NexusFlow？

单Agent需要同时维持以下认知负荷：
- **工作记忆**: 6,488个有效数据点的统计特征
- **领域知识**: 10个国家各自的经济结构特征
- **分析框架**: 4维度 × 5指标的综合评分模型
- **交叉验证**: 指标间逻辑一致性（如储蓄-投资恒等式）
- **时间推理**: 历史趋势 + 预测值的双重解读

这些维度同时作用，远超单Agent的有效认知带宽。

### 3.2 各Agent在宏观经济任务中的具体职责

#### 🎯 Coordinator（编排者）— 全局调度中枢

**在本任务中的职责**：
1. **阶段调度**: 管理5个分析阶段的流水线——数据采集→单维分析→交叉验证→综合评分→报告生成
2. **国家间矛盾仲裁**: 当Researcher发现中国储蓄率(NGSD_NGDP~45%)与消费不足矛盾时，协调Artisan和Assayer给出不同视角
3. **负载均衡**: 将10国的并行分析任务均匀分配，避免单个Agent过载
4. **降级策略**: 俄罗斯(RUS)数据缺失率37.3%时，调整分析深度

**CDoL视角**: Coordinator拥有**全局视图**，可读取所有Agent的CDoL分区，但不直接操作数据。它通过Handoff协议调度任务流转。

#### 📋 Planner（规划者）— 策略推理核心

**在本任务中的职责**：
1. **任务树分解**: 将"宏观经济韧性评估"分解为：
   ```
   T1: [Researcher] 拉取15指标×10国数据 ← DBnomics
   T2: [Executor] 数据清洗与标准化 ← T1
   T3: [Executor] 4维度评分计算 ← T2
   T4: [Artisan] 各国特殊因素标注 ← T2
   T5: [Assayer] 跨指标一致性检验 ← T3
   T6: [Miner] IMF WEO原文对标 ← T3
   T7: [Reviewer] 评分结果合理性审查 ← T3,T4,T5
   T8: [Caster] 生成热力图/雷达图 ← T7
   T9: [Archivist] 经验蒸馏为Skill Card ← T1-T8
   ```
2. **策略选择**: 决定使用"维度优先"而非"国家优先"的分析路径
3. **风险预判**: 标注俄罗斯/中国数据缺失风险，提前规划降级方案

**CDoL视角**: Planner写入`plan`分区，所有Agent可读。Executor只读`task_assignments`分区。

#### 🔍 Researcher（研究者）— 知识获取核心

**在本任务中的职责**：
1. **数据检索**: 通过DBnomics CLI拉取IMF WEO全部150条时间序列
2. **数据验证**: 检查6,488个数据点的完整性——标记NA模式（俄罗斯2022年后数据缺失）
3. **多源交叉**: 比对IMF WEO的政府债务数据(GGXWDG_NGDP)与IMF IFS的对应指标
4. **异常检测**: 标记异常值——如2020年各国GDP增速骤降、巴西通胀异常

**输出**: 带置信度标注的结构化数据集（存入`data`分区）

**CDoL视角**: Researcher写入`data`分区和`verification`分区。其他Agent只读`data`，Assayer可读写`verification`。

#### ⚡ Executor（执行者）— 精确执行核心

**在本任务中的职责**：
1. **数据清洗**: 处理NA值——前向填充/插值/标记缺失模式
2. **Z-score标准化**: 对15个指标做跨国家标准化
   ```python
   z_score = (x - mean_country) / std_country
   ```
3. **维度评分**: 按4维度加权计算综合分数
   - 经济增长维度: GDP增速(0.5) + 人均GDP(0.3) + 产出缺口(0.2)
   - 价格稳定维度: CPI通胀(0.6) + 失业率(0.4)
   - 财政健康维度: 净债务(-0.3) + 收支余额(0.3) + 总收入(0.2) + 总支出(-0.2)
   - 外部平衡维度: 经常账户(0.3) + 储蓄率(0.3) + 投资率(0.2) + 贸易增速(0.2)
4. **趋势分析**: 计算2010-2024年CAGR、波动率、趋势斜率

**CDoL视角**: Executor写入`computed`分区。Reviewer可读写`quality_flags`分区。

#### 🔎 Reviewer（审查者）— 质量守门员

**在本任务中的职责**：
1. **评分逻辑审查**: 检查维度权重是否合理（如财政健康维度中债务是否为负向指标）
2. **国家排名合理性**: 验证排名是否符合经济学直觉
   - 德国应排名前列（低债务、高出口）
   - 日本应排名靠后（高债务/GGDP>200%）
   - 印度应排名靠后（高通胀、高失业率）
3. **数据异常标记**: 标记可能的数据质量问题
   - 中国GDP增速与用电量/货运量不匹配
   - 俄罗斯2022年后数据可信度
4. **一致性检查**: 验证"储蓄-投资 = 经常账户"恒等式在数据中的成立程度

**CDoL视角**: Reviewer写入`quality_flags`和`review_notes`分区。Coordinator和Planner可读。

#### ⛏️ Miner（矿工）— 文献挖掘专家

**在本任务中的职责**：
1. **IMF WEO原文检索**: 拉取IMF对各国最新Article IV咨询报告的关键结论
2. **学术文献对标**: 检索主权债务可持续性分析(DSA)框架文献
3. **风险模型提取**: 从学术论文中提取早期预警指标(EWI)的阈值
   - 债务/GDP > 90% (Reinhart-Rogoff阈值)
   - 经常账户逆差/GDP > 5% (金融危机预警)
   - CPI通胀 > 40% (恶性通胀边界)
4. **历史案例库**: 整理过去20年重大经济危机的前兆指标模式

**CDoL视角**: Miner写入`literature`和`thresholds`分区。Artisan和Assayer可读。

#### 🔬 Assayer（试金）— 知识验证专家

**在本任务中的职责**：
1. **跨数据源一致性**: 
   - IMF WEO的政府总债务(GGXWDG_NGDP) vs IMF IFS的对应数据
   - 中国官方失业率(LUR) vs 其他来源估算
2. **内部一致性检验**:
   - 政府总收入(GGR_NGDP) - 政府总支出(GGX_NGDP) ≈ 财政收支余额(GGXCNL_NGDP)
   - 国民储蓄(NGSD_NGDP) - 总投资(NID_NGDP) ≈ 经常账户(BCA_NGDPD)
3. **异常值检测**: 
   - 使用IQR方法标记离群值
   - Z-score > 3的数据点标记为⚠️
4. **时间一致性**: 检查预测值(2025-2030)与历史趋势的偏离程度

**CDoL视角**: Assayer读写`verification`分区，写入`anomalies`分区。Reviewer可读。

#### 🔨 Caster（铸师）— 代码生成专家

**在本任务中的职责**：
1. **热力图代码**: 生成10国×15指标的Z-score热力图（Python/matplotlib）
2. **雷达图代码**: 生成每国4维度评分的雷达图对比
3. **趋势图代码**: 生成关键指标( GDP增速、通胀、债务)的2010-2029时间序列对比图
4. **交互式Dashboard**: 生成HTML+JS的交互式数据探索界面

**输出规范**: 所有代码包含完整import、注释、错误处理，可直接运行。

**CDoL视角**: Caster写入`code`分区。Executor可读取`code`执行。

#### 🎨 Artisan（匠人）— 领域专家

**在本任务中的职责**：
1. **国别特殊因素标注**:
   - 🇷🇺 俄罗斯: 2022年制裁后GDP数据可信度下降，石油价格依赖
   - 🇨🇳 中国: 房地产债务危机(2021-)、地方政府隐性债务、人口结构拐点
   - 🇯🇵 日本: 长期通缩、超宽松货币政策、政府债务/GDP>260%的特殊性
   - 🇧🇷 巴西: 通胀历史(2015年CPI>10%)、政治周期对财政的影响
   - 🇮🇳 印度: 高非正规经济占比、卢比贬值压力、结构性通胀
   - 🇿🇦 南非: 高失业率(>30%)、电力危机、矿业依赖
2. **指标解读**: 
   - 产出缺口(NGAP_NPGDP)仅251个有效点(32.8%覆盖率)，解释缺失原因
   - 政府净债务(GGXWDN_NGDP)仅277点，因为许多国家只报告总债务
3. **风险定级**: 基于领域知识为每个国家的每个维度给出风险等级(低/中/高/极高)

**CDoL视角**: Artisan写入`domain_knowledge`分区。Planner和Reviewer可读。

## 四、CDoL信息策略设计

### 4.1 CDoL分区规划

针对本任务，10个Agent的CDoL（Cognitive Division of Labor）分区如下：

| 分区名 | 写入者 | 读者 | 内容 |
|--------|--------|------|------|
| `plan` | Planner, Coordinator | 全部 | 任务树、阶段计划 |
| `data` | Researcher | 全部(只读) | 原始数据、数据验证报告 |
| `computed` | Executor | Reviewer, Caster, Archivist | 标准化数据、评分结果 |
| `quality_flags` | Reviewer, Assayer | Coordinator, Planner | 质量标记、异常报告 |
| `domain_knowledge` | Artisan | Planner, Reviewer, Assayer | 国别特殊因素、风险定级 |
| `literature` | Miner | Artisan, Assayer, Planner | 文献发现、阈值标准 |
| `verification` | Assayer, Researcher | Reviewer | 交叉验证结果 |
| `code` | Caster | Executor | 可视化/计算代码 |
| `memory` | Archivist | 全部(只读) | Skill Card、经验索引 |
| `arbitration` | Coordinator | 相关Agent | 冲突仲裁决议 |

### 4.2 信息流图

```
Phase 1: 数据采集阶段
  Researcher ──data──→ [共享数据池]
  Researcher ──verification──→ Assayer

Phase 2: 并行分析阶段 (扇出)
  ┌── Executor ←──plan── Planner ──→ computed
  ├── Artisan ←──data── → domain_knowledge
  ├── Miner ←──data── → literature
  └── Assayer ←──data── → verification

Phase 3: 交叉验证阶段
  Assayer ──quality_flags──→ Reviewer
  Reviewer ←──domain_knowledge── Artisan
  Reviewer ←──literature── Miner

Phase 4: 综合与可视化
  Caster ←──computed── Executor → code
  Executor ←──code── Caster (执行可视化)

Phase 5: 归档与蒸馏
  Archivist ←──全部── → memory (Skill Card)
```

### 4.3 信息隔离规则

| 规则 | 说明 | 理由 |
|------|------|------|
| Executor不能修改data分区 | 原始数据只读 | 防止计算污染数据源 |
| Artisan不能修改computed分区 | 领域知识不影响计算结果 | 分离客观计算与主观解读 |
| Miner不直接修改quality_flags | 文献证据通过Assayer转化 | 间接证据需要验证环节 |
| Caster只读computed | 铸师不修改分析结果 | 代码生成与分析解耦 |
| Reviewer可以回退Executor的结果 | 质量不通过则重做 | 质量门控的权威性 |

## 五、5阶段执行流程

### Phase 1: 数据采集与验证 (T1-T2)

```
Coordinator → Researcher: "从DBnomics拉取IMF WEO全部数据"
Researcher → data: 6,488个有效数据点 + 7,650个数据单元格
Researcher → Assayer: "请验证数据完整性"
Assayer → verification: 覆盖率报告，标记NA模式
```

## 六、数据量充分性论证

### 6.1 认知负荷量化

**单Agent需要同时处理的信息量**：

| 信息类型 | 数据量 | 认知负荷 |
|----------|--------|---------|
| 原始数据 | 6,488个有效数据点 | 需要记忆各国各指标的统计特征 |
| 指标关系 | C(15,2)=105对指标关系 | 需要判断哪些关系需要验证 |
| 国家特征 | 10国 × 3+特殊因素 = 30+条领域知识 | 需要同时维持所有国别背景 |
| 时间推理 | 51年 × 10国趋势 + 20年预测 | 需要识别趋势拐点和结构变化 |
| 评分计算 | 4维度 × 10国 × 加权公式 | 需要确保权重一致性 |
| **总认知负荷** | **~6,500数据点 + 105关系 + 30+知识** | **远超7±2工作记忆限制** |

### 6.2 为什么单Agent会失败？

1. **注意力稀释**: 6,488个数据点要求Agent在长上下文中精确定位信息。研究表明LLM在>4K token的有效工作窗口内精度显著下降（"Lost in the Middle"问题）。

2. **角色冲突**: 同一Agent需要同时扮演：
   - 数据分析师（客观计算）
   - 领域专家（主观解读）
   - 质量审查（自我纠错）
   
   这种角色冲突导致"确认偏误"——Agent倾向于验证自己的分析而非挑战它。

3. **串行瓶颈**: 单Agent必须串行处理6,488个数据点，而10个Agent可以并行处理（每Agent约649个点）。

4. **上下文窗口消耗**: 仅原始数据就需要 ~20K tokens（6,488 × 3 tokens/number），加上分析框架、领域知识、输出模板，轻松超过100K tokens。

### 6.3 NexusFlow 10-Agent优势

| 维度 | 单Agent | NexusFlow 10-Agent |
|------|---------|-------------------|
| 有效处理数据量/Agent | 6,488 | ~649 (降10倍) |
| 并行度 | 1 (串行) | 4-6 (扇出阶段) |
| 角色纯度 | 混合(分析+审查+解读) | 纯(每个Agent单一职责) |
| 错误检测 | 自检(确认偏误) | 交叉审查(Assayer+Reviewer) |
| 知识利用 | 上下文内(有限) | 分区存取(无上限) |
| 经验复用 | 无 | Skill Card持久化 |

### 6.4 实验假设

**H1**: NexusFlow 10-Agent的评分结果与IMF官方评估的相关系数 > 0.85

**H2**: 10-Agent方案的数据异常检出率 > 单Agent方案的2倍

**H3**: 10-Agent方案的国家排名稳定性（bootstrap重采样）> 单Agent方案

**H4**: 通过CDoL信息策略，Agent间的信息传递效率 > 全共享模式的1.5倍（减少噪声干扰）

## 十、结论

本设计方案基于DBnomics真实拉取的**6,488个有效宏观经济数据点**（远超v1估算的1,280），完整映射了NexusFlow全部**10个Agent**在"全球宏观经济韧性评估"任务中的具体职责、信息流和协作模式。

核心论点：
1. **数据量充分**: 6,488有效数据点 × 105对指标关系 × 10国领域知识 = 单Agent无法有效处理的认知负荷
2. **10-Agent必要**: 每个Agent有明确的不可替代角色（Researcher拉数据、Executor计算、Artisan解读、Assayer验证、Archivist蒸馏）
3. **CDoL有效**: 信息分区策略确保Agent只关注相关子集，避免信息过载
