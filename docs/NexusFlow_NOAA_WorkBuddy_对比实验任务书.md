## §1 背景：NexusFlow框架是什么

### 1.1 一句话定位

**NexusFlow** 是一个面向超长程复杂任务的**动态异构群体智能架构**（Dynamic Heterogeneous Swarm Intelligence Architecture）。它不是一个更大的LLM，而是一套让多个异构Agent高效协作的**协议框架**。

### 1.2 核心理念

**框架工程 > 模型堆叠**。NexusFlow 的核心信念是：复杂任务的瓶颈不在于单个模型有多强，而在于如何组织多个专业化Agent的信息流、决策流和纠错流。类比来说——NexusFlow 不是造一个更强的士兵，而是设计一套让10个不同专长的士兵高效协同作战的指挥系统。

### 1.3 核心机制：CDoL（Cognitive Data Orchestration Layer）

**CDoL（认知数据编排层）** 是 NexusFlow 独创的信息流控制机制，也是区别于其他多Agent框架的关键差异点。

**CDoL 解决的核心问题**：在多Agent协作中，每个Agent应该看到什么数据？看到太多会信息过载、角色混淆；看到太少会遗漏关键信息、决策偏颇。

**CDoL 的运作方式**：
- **角色感知数据切片**：每个Agent只能访问与自己角色定义相关的数据子集。例如 Researcher 负责获取原始数据，Executor 负责基于 Researcher 提供的清洗后数据执行分析，Reviewer 只看最终输出和中间结论。
- **全局共识度（Global Consensus Score）**：一个0-1的浮点数，表示当前所有Agent对某个决策点的共识程度。每轮迭代后更新，低于阈值（默认0.7）的议题必须继续讨论。
- **上下文传播协议**：Agent之间的信息传递不是简单的消息广播，而是经过CDoL编排的结构化上下文包（Context Packet），包含：任务目标摘要、前序Agent的关键结论、相关数据子集、当前共识度。

### 1.4 10个Agent的角色定义

NexusFlow 的10个Agent分为两种层级：

- **PRO（专业级）**：处理需要深度推理的复杂任务，运行在CLOUD层
- **FLASH（快速级）**：处理高频、标准化的执行任务，运行在EDGE层（延迟更低）

| Agent | 角色 | 核心职责 | Model Tier | Run Layer |
|---|---|---|---|---|
| **Coordinator** | 多Agent协作调度中枢 | 负责任务分解、Agent调度、冲突仲裁、进度监控。是整个系统的"大脑中的大脑"。 | PRO | CLOUD |
| **Planner** | 策略推理核心 | 负责将模糊的高层目标转化为可执行的步骤序列。处理"做什么"和"为什么做"。 | PRO | CLOUD |
| **Researcher** | 知识获取与验证核心 | 负责从数据源获取原始信息，进行初步清洗和结构化。处理"数据从哪来"。 | FLASH | EDGE |
| **Executor** | 精确执行核心 | 按照Planner的步骤序列，执行具体的计算、分析、生成操作。处理"怎么做"。 | FLASH | EDGE |
| **Reviewer** | 质量守门员 | 对所有Agent的输出进行质量审查，标记不一致、错误或遗漏。处理"做得对不对"。 | FLASH | EDGE |
| **Miner** | 文献挖掘专家 | 从海量数据/文档中提取关键模式、趋势、异常。擅长发现隐藏相关性。 | PRO | CLOUD |
| **Assayer** | 知识验证专家 | 对Miner/Researcher的发现进行独立交叉验证。用不同方法验证同一结论。 | FLASH | CLOUD |
| **Caster** | 代码生成专家 | 将分析需求转化为可执行代码（Python/R/SQL），并执行计算密集型任务。 | PRO | CLOUD |
| **Artisan** | 领域专家 | 提供气候科学的领域知识，判断分析结果的气象学合理性。 | FLASH | EDGE |
| **Archivist** | 知识守护者与记忆管理者 | 管理长期记忆、维护知识库、确保信息不丢失不重复。 | FLASH | EDGE |

### 1.5 信息流协议（简化版）

```
[用户任务] 
    → Coordinator（分解任务）
        → Planner（制定分析策略）
            → Researcher + Miner（并行获取数据与文献）
                → Assayer（交叉验证发现）
                    → Executor + Caster（执行计算与分析）
                        → Artisan（领域知识审核）
                            → Reviewer（质量审查）
                                → Archivist（归档结论）
    → Coordinator（汇总共识度，决定是否需要迭代）
        → 共识度 ≥ 0.7 → 输出最终结果
        → 共识度 < 0.7 → 回到Planner重新规划
```

## §2 实验目标

### 2.1 核心实验问题

> **在同一个复杂的全球气象分析+气候预测任务上，10Agent NexusFlow 协作模式相比单Agent独立模式，在分析质量、预测精度、洞察深度等维度上是否存在显著优势？优势的具体来源是什么？**

### 2.2 实验组设计

| 组别 | 模式 | 描述 |
|---|---|---|
| **A组（基线）** | 单Agent | 一个LLM Agent独立完成全部L1+L2分析任务。可以类比为一个"全能分析师"——没有角色切换、没有协作协议、没有共识度机制。 |
| **B组（实验）** | NexusFlow 10Agent | 10个Agent严格按§1.4的角色定义和§1.5的信息流协议协作完成任务。可以由一个LLM扮演多个角色（因为实验环境限制），但**必须严格遵循角色协议和信息流约束**——即每个"角色"只能访问其CDoL授权的数据子集，必须通过结构化Context Packet传递信息。 |

### 2.3 关键控制变量

- **同一个LLM底座**：两组使用相同的底层模型（如GPT-4o、Claude 3.5 Sonnet等），确保差异来自框架而非模型能力。
- **同一份数据**：两组看到完全相同的数据（表A+表B），但NexusFlow组的数据访问经过CDoL编排。
- **同一个任务Prompt**：L1任务的指令文本完全一致（见§4），区别只在于"一个Agent做"还是"10个Agent分工做"。

## §4 双层任务设计

### 4.1 L1 历史分析任务（基于表A，2015-2022）

#### 任务Prompt（两组完全一致）

```
你是一位受世界气象组织（WMO）委托的高级气候分析师。请基于2015-2022年27个全球主要城市的5项气象指标月度数据，完成以下分析：

1. 气候趋势分析（Climate Trend Analysis）
   对每个城市，分析2015-2022年间各气象指标的时间趋势：
   - 识别是否存在显著的升温/降温趋势（TMAX、TMIN、TAVG的年均值变化方向和幅度）
   - 识别降水模式变化（PRCP的年内分布是否发生偏移，如雨季提前/延后）
   - 计算每个城市的年平均气温和年总降水量，给出8年间的变化趋势
   - 对于有积雪数据的城市（Toronto、Moscow、Berlin等），分析积雪季节长度和深度的变化
   - 明确指出哪些城市数据覆盖率不足、趋势结论需标注为"低置信度"

2. 城市对比分析（Cross-City Comparison）
   将27个城市按气候类型分组对比：
   - **热带组**（Bangkok, Jakarta, Singapore, Ho Chi Minh, Manila, Dhaka, Karachi, Lagos, Lima, Nairobi）
   - **温带/亚热带组**（Beijing, Tokyo, Delhi, Busan, Mexico City, Sao Paulo）
   - **地中海/干旱组**（Cairo, Madrid, Istanbul, Dubai）
   - **大陆性气候组**（Moscow, Berlin, Paris）
   - **温带海洋/其他组**（Buenos Aires, Sydney, Santiago, Toronto）
   
   对比维度：
   - 年均温差（TMAX年均 - TMIN年均），反映大陆性程度
   - 降水集中度（降水量最高的3个月占全年比例）
   - 季节对称性（南北半球城市的季节差异）
   - 给出每组内"气候最极端"和"气候最温和"城市的排名及依据

3. 异常年份识别（Anomaly Year Detection）
   识别2015-2022年间的异常气候事件：
   - 对每个城市、每个指标，计算8年间的月均值分布，标记偏离均值±2σ的月份为异常
   - 汇总哪些年份出现了最多异常事件（如2019是否是全球性高温年？）
   - 识别跨城市同步异常：哪些异常同时影响了多个城市？（如El Niño/La Niña的全球性影响）
   - 对每个异常事件，给出可能的驱动因素分析（如"2020年Delhi降水异常偏高可能与当年强季风有关"）

4. 区域气候模式识别（Regional Climate Pattern Discovery）
   发现至少5组跨区域的气候关联模式：
   - 例如：同纬度城市的TMAX年变化相关性
   - 例如：El Niño年份对太平洋两岸城市（Tokyo-Busan, Manila-Singapore）的不同影响
   - 例如：大陆性气候城市（Moscow-Berlin-Paris）的温度梯度变化
   - 例如：热带城市（Bangkok-Dhaka-Delhi）季风季节的降水同步性
   - 每组模式需说明：涉及哪些城市、哪些指标、相关性强弱、时间窗口、可能的气象学机制
```

### 4.2 L2 预测回测任务（对照表B真值）

在Agent完成L1任务后，额外给出如下预测任务（可融入L1任务4中，也可作为独立环节）：

```
5. 短期气候预测（Short-term Climate Forecasting）
   基于2015-2022年的历史数据，对每个有数据的城市预测2023-2024年：
   - 各月份的TMAX、TMIN、TAVG（给出预测区间，如"2023年7月TMAX预测：34.5°C-37.2°C"）
   - 各月份的PRCP（给出预测区间）
   - 年度总降水量和年平均气温的预测
   
   每个预测需包含：
   - 预测区间（置信区间）
   - 置信度（高/中/低）
   - 预测方法简述（如"基于线性趋势外推"或"基于季节性ARIMA模型"或"基于气候学平均值"）
   - 主要不确定性来源
```

#### 回测指标

| 指标 | 计算方式 | 说明 |
|---|---|---|
| **区间命中率（Hit Rate）** | 预测区间包含真实值的比例 | 在有真值的城市×月份×指标中，有多少被预测区间覆盖 |
| **MAE** | mean(\|预测中值-真实值\|) | 平均绝对误差（温度单位°C，降水单位mm），越低越好 |
| **MAPE** | mean(\|预测中值-真实值\|/\|真实值\|)×100% | 平均绝对百分比误差，适用于温度指标 |
| **区间宽度（Interval Width）** | 预测区间上界-下界的平均值 | 越窄越好（但需保证命中率≥60%） |
| **驱动因素准确度** | 人工评估 | 对异常事件驱动因素分析的合理性打分（1-5分），抽样评估10个城市 |

#### 回测计算方法

1. **温度指标（TMAX/TMIN/TAVG）**：
   - 对每个有真值的城市-月份，检查真实值是否落在预测区间内
   - 计算 MAE（°C）和 MAPE（%）
   
2. **降水指标（PRCP）**：
   - 降水数据波动大，MAPE在接近0时失真，改用MAE（mm）
   - 区间命中率单独计算
   
3. **积雪指标（SNWD）**：
   - 仅有Toronto有显著积雪数据，单独评估
   
4. **汇总统计**：
   - 按气候分组汇总命中率
   - 按数据覆盖率分层（高覆盖>50%、中覆盖20-50%、低覆盖<20%）分析预测质量差异

## §5 10个Agent的具体分工映射

### 5.1 气象分析场景下的Agent职责

| Agent | 在NOAA气象分析中的具体任务 | CDoL数据访问权限 |
|---|---|---|
| **Coordinator** | 分解L1的4个子任务和L2预测任务；调度各Agent执行顺序；仲裁不同Agent对趋势判断的分歧（如Miner认为全球变暖vs Artisan认为是自然波动） | 仅访问任务分解表和进度状态 |
| **Planner** | 制定分析策略：先做数据质量评估→再做趋势分析→再做对比→再做异常检测→最后预测；确定各子任务的优先级和依赖关系 | 访问数据总览sheet + 覆盖率统计 |
| **Researcher** | 理解数据结构和含义；识别缺失数据的分布模式；标记覆盖率不足的城市/指标/时段；提供数据质量报告给后续Agent | 访问全部数据表但只提取质量统计信息 |
| **Executor** | 执行具体的统计计算：年均值、标准差、趋势斜率、异常标记、相关系数等；按照Planner的步骤逐个计算 | 访问温度/降水/积雪数据表（已清洗版本） |
| **Reviewer** | 审查Executor的计算结果是否合理：温度值是否在合理范围（如-50°C至+60°C）、降水值是否为非负、趋势方向是否与数据一致 | 访问最终输出 + 抽样原始数据 |
| **Miner** | 从数据中发现隐藏模式：如"东京2019年夏季TMAX异常偏高与当年热浪事件的关联"、"Toronto积雪季节逐年缩短"等 | 访问全部数据表（需要全局视野发现跨表模式） |
| **Assayer** | 交叉验证Miner的发现：用不同方法验证（如用线性回归和Mann-Kendall检验分别验证趋势显著性）；验证跨城市相关性是否稳健 | 访问Miner的结论 + 相关城市的原始数据子集 |
| **Caster** | 编写分析代码：趋势拟合代码、异常检测算法、相关矩阵计算、预测模型代码；确保计算可复现 | 访问数据子集 + 输出代码和计算结果 |
| **Artisan** | 提供气候学领域知识：判断趋势是否合理（如"Cairo升温趋势是否与撒哈拉沙漠扩张一致"）；解释气象现象（如"El Niño对太平洋城市的影响"） | 访问Miner/Assayer的验证结论 |
| **Archivist** | 记录所有分析结论、中间结果、数据质量问题；确保不重复分析；维护城市气候档案 | 访问所有已产出的结论和报告 |

## §6 评估维度（必须全部评估）

### 6.1 六维评估框架

| 维度 | 编号 | 权重 | 评估对象 | 评分标准（1-10分） |
|---|---|---|---|---|
| **完整性** | D1 | 15% | L1输出 | 10分=覆盖全部27城市×5指标，无遗漏；每遗漏1城市扣0.37分，每遗漏1组指标扣0.5分；正确标注数据不足的城市 |
| **准确率** | D2 | 20% | L1任务1+2+L2回测 | 10分=趋势方向/幅度与数据一致，无事实性错误；5分=大体方向正确但有2-3处偏差；1分=多处与数据矛盾 |
| **效率** | D3 | 10% | A+B组 | 10分=NexusFlow组在3轮内完成分析且总Token消耗<单Agent的1.5倍；5分=5轮收敛；1分=发散未收敛或Token消耗>3倍 |
| **协作深度** | D4 | 15% | 仅B组 | 10分=≥5次有意义的跨Agent交互（修正/补充/质疑）；7分=3-4次；3分=仅形式化交互无实质内容 |
| **洞察质量** | D5 | 25% | L1任务3+4 | 10分=≥6组高质量跨区域模式，有气象学机制解释和统计验证；7分=4-5组合理规律；3分=仅1-2组弱相关 |
| **可复现性** | D6 | 15% | A+B组 | 10分=3次运行结果一致率>90%且所有计算有代码支撑；7分=一致率>70%；3分=一致率<50%或无可执行代码 |

### 6.2 协作效率详细指标（仅NexusFlow组）

#### 6.2.1 共识度收敛曲线

- **定义**：每轮迭代后，所有参与Agent对当前结论的共识度评分（0-1）
- **采集方式**：每轮迭代结束时，Coordinator汇总各Agent的结论一致性评分
- **期望形态**：初始共识度0.3-0.5 → 2-3轮后收敛至0.7-0.9
- **绘图要求**：X轴=迭代轮次，Y轴=全局共识度，曲线应展示收敛趋势

#### 6.2.2 信息流效率

- **定义**：CDoL编排下的有效数据访问次数 / 单Agent模式下的总数据访问次数
- **计算**：统计NexusFlow模式下各Agent实际访问的数据行数，与单Agent一次性加载全部数据的对比
- **期望**：NexusFlow的总数据访问量 ≤ 单Agent（因为CDoL做了数据切片，避免冗余访问）

#### 6.2.3 错误自我修正率

- **定义**：Reviewer/Assayer在协作过程中发现并成功纠正的错误数 / 总错误数
- **采集方式**：记录每轮Reviewer的审查意见和修正建议
- **期望**：修正率 > 30%（即10个错误中至少修正3个）

## §8 关键约束

### 8.1 数据真实性

- **严禁编造数据**：所有分析必须基于表A/表B中的真实数据点
- 对于缺失数据，可以说明"该城市该时段数据缺失，无法分析"，但不可编造替代值
- 所有统计结论必须可追溯到原始数据
- 数据覆盖率不足的城市（<20%），结论必须标注为"低置信度"

### 8.2 NexusFlow模式忠实度

- **必须严格体现10个Agent的角色分工**：不能出现"Agent A做了Agent B的工作"
- **必须严格体现信息流协议**：不能出现"所有Agent同时看到所有数据"
- **必须体现CDoL的数据切片机制**：不同Agent看到的数据子集应有所不同
- 如果由一个LLM扮演多个角色，必须通过System Prompt切换来模拟角色差异

### 8.3 单Agent基线纯洁性

- 单Agent必须是"一个Agent做所有事"，不做任何内部角色拆分
- 可以一次读取全部数据，无需进行数据切片
- 没有共识度机制，没有迭代纠错（除非Agent自发修正）

### 8.4 预测规范

- 所有预测结果必须明确标注**置信区间**（如"28.5°C-31.2°C"）
- 必须标注**置信度**（高/中/低）
- 区间不能过宽（如"-10°C-50°C"视为无效预测）
- 区间不能过窄（如"30.01°C-30.02°C"视为过拟合）
- 对于覆盖率<20%的城市，预测置信度必须标注为"低"

### 8.5 气象学合理性

- 温度值必须在合理范围内（地球表面气温不可能<-90°C或>+60°C）
- 降水量必须为非负值
- TAVG应大致介于TMAX和TMIN之间
- 热带城市的年均温应>20°C，大陆性气候城市应有显著季节温差
- 如果分析结果违反上述常识，需要检查是数据问题还是分析错误

### 8.6 报告质量

- 实验报告需适合给**比赛评委**阅读
- 专业但不晦涩——假设读者了解AI/LLM，但不了解NexusFlow细节
- 所有NexusFlow专有术语首次出现时需解释
- 图表需有清晰的标题和注释

### 10.2 常见陷阱

1. **不要让NexusFlow模式退化为"单Agent换了10个名字"**——必须有真实的信息流约束和角色差异
2. **不要忽略缺失数据**——67.8%的缺失率（表A）是真实世界气象API数据的特征，分析中应体现对缺失数据的处理策略，而非简单忽略
3. **不要编造统计数字**——如果无法精确计算相关系数，可以给出定性判断（强正相关/弱负相关等），但要标注
4. **不要跳过L2回测**——这是验证预测质量的关键步骤，也是D2评分的依据
5. **共识度曲线不要太完美**——真实的共识度收敛通常有波动，不是完美的指数衰减
6. **注意数据单位**——温度已经是°C（原始数据已除以10），降水已经是mm（原始数据已除以10），积雪已经是mm
7. **注意Santiago覆盖率为0%**——该城市在表A和表B中均无数据，分析中应明确排除并说明原因
8. **SNWD指标仅少数城市有值**——仅Toronto（100%）、Tokyo（7.3%）、Moscow（13.5%）等少数城市有积雪数据，大多数城市SNWD全为0或空
