# 方案一：认知分工协同推理引擎 (Cognitive Division of Labor Engine)

> 基于用户原创设计文档，结合 nexusflow 现有框架的落地方案

---

## 核心命题

协同推理的增益不来自"多模型ensemble的统计降噪"，而来自**"在信息受限条件下被迫完成的认知过程"**——任何单体Agent（即使拿到完整信息）都无法独立达到的推理深度。

这与 Phase 6 的 DynamicTopologyRouter 的关系：Router 决定"谁来推理+怎么分配视角"，CDoL Engine 决定"推理过程中如何通信+如何融合"。Router 是骨架，CDoL 是大脑。

## 二、组件设计

### 2.1 PerspectiveDecomposer（视角分解器）

**输入**：原始问题 Q + 可用 Agent 能力画像（来自 DynamicTopologyRouter 的 AgentCapabilityProfile）

**输出**：N 个非对称视角分配方案

```python
@dataclass
class PerspectiveAssignment:
    """单个视角分配"""
    agent_id: str
    perspective_question: str      # 该Agent看到的子问题
    context_mask: ContextMask      # 上下文掩码（该Agent能看到什么、不能看到什么）
    role_constraint: Optional[str] # 角色约束（如"质疑者"/"辩护者"）
    resource_subset: List[str]     # 分配给该Agent的资源子集

@dataclass
class ContextMask:
    """上下文掩码——主动不对称的核心"""
    allowed_evidence: List[str]       # 允许看到的证据ID
    blocked_evidence: List[str]       # 显式屏蔽的证据ID
    allowed_domains: List[str]        # 允许访问的知识域
    blocked_domains: List[str]        # 屏蔽的知识域
    abstraction_level: str            # "concrete" | "abstract" | "mixed"
    time_slice: Optional[Tuple]       # 时间切片范围（如有）

@dataclass
class DecompositionPlan:
    """视角分解方案"""
    input_question: str
    strategy: str                     # 使用的分解策略
    assignments: List[PerspectiveAssignment]
    rationale: str                    # 为什么这样分解
    bridgeability_score: float        # 视角间的可桥接性评分（0-1）
```

**六种分解策略的实现**：

| 策略 | 实现方式 | 示例（科研场景） |
|------|---------|----------------|
| 证据拆分 | 将检索到的文献分为互补子集 | Agent₁看实验数据，Agent₂看理论推导 |
| 角色约束 | 在prompt中注入对抗角色 | Agent₁"质疑假设"，Agent₂"辩护假设" |
| 层级分离 | 抽象层 vs 实施层 | Agent₁做方法论设计，Agent₂做可行性验证 |
| 模态拆分 | 结构化 vs 非结构化 | Agent₁看数值数据表，Agent₂看论文文本描述 |
| 时间切片 | 按时间分割序列数据 | Agent₁看早期研究，Agent₂看最新进展 |
| 抽象层级 | 具体实例 vs 抽象规则 | Agent₁看案例，Agent₂看理论框架 |

**自动化生成**：视角分解器本身是一次轻量级 LLM 调用。

```python
class PerspectiveDecomposer:
    def decompose(self, question: str, agents: List[AgentCapabilityProfile]) -> DecompositionPlan:
        """
        自动生成视角分配方案。
        
        核心prompt结构：
        1. 描述问题Q的推理维度
        2. 列出可用Agent的能力画像
        3. 要求选择分解策略，并生成非对称视角分配
        4. 评估bridgeability_score（太低=视角差距过大无法桥接）
        """
        prompt = self._build_decomposition_prompt(question, agents)
        plan_json = await self.llm.call(prompt, model="flash")  # 轻量调用
        plan = self._parse_plan(plan_json)
        
        # 自检：bridgeability不能太低（否则通信无法弥合差距）
        if plan.bridgeability_score < 0.3:
            plan = self._fallback_to_evidence_split(plan)
        
        return plan
```

### 2.2 CommunicationLayer（有损通信层）

**核心约束**：
- ✅ 允许传递：中间结论 C、置信度、关键假设、排除的假设列表
- ❌ 禁止传递：原始视角 Qᵢ、上下文掩码内容、"我看到了什么"

```python
@dataclass
class IntermediateConclusion:
    """Agent产出的中间结论——通信的唯一载体"""
    agent_id: str
    conclusion: str                    # 推理结论
    confidence: float                  # 置信度 0-1
    reasoning_chain: List[Step]        # 推理链（结构化）
    active_hypotheses: List[str]       # 当前活跃假设
    eliminated_hypotheses: List[str]   # 已排除的假设及排除理由
    key_assumptions: List[str]         # 关键假设（可能被挑战）
    uncertainty_markers: List[str]     # 不确定性标记

@dataclass
class Step:
    """推理链中的单步"""
    step_id: int
    operation: str        # "infer" | "exclude" | "confirm" | "question"
    input: str            # 这一步的输入
    output: str           # 这一步的输出
    basis: str            # 依据（引用证据或逻辑规则）

@dataclass
class DifferenceAttribution:
    """Agent在Round 1产出的差异归因"""
    my_conclusion: str
    other_conclusions: List[str]
    contradictions: List[Contradiction]    # 发现的矛盾
    attributions: List[Attribution]        # 对矛盾的归因解释
    revision: Optional[str]                # 修正后的结论
    revision_reason: str                   # 修正原因

@dataclass
class Contradiction:
    """矛盾"""
    my_position: str
    other_position: str
    severity: float           # 矛盾严重程度
    attributable: bool        # 能否从视角差异解释

@dataclass
class Attribution:
    """差异归因"""
    contradiction_id: int
    explanation: str          # "对方可能拥有X信息，导致其结论偏向Y"
    inferred_other_context: str  # 推断对方可能看到的上下文
```

**通信协议实现**：

```python
class CommunicationLayer:
    """有损通信层——认知分工的理论核心"""
    
    async def run_round_0(self, assignments: List[PerspectiveAssignment]) -> List[IntermediateConclusion]:
        """Round 0: 各Agent独立推理，无通信"""
        tasks = []
        for assignment in assignments:
            tasks.append(self._agent_reason(assignment))
        return await asyncio.gather(*tasks)
    
    async def run_round_1(self, conclusions: List[IntermediateConclusion], 
                          assignments: List[PerspectiveAssignment]) -> List[DifferenceAttribution]:
        """
        Round 1: 差异归因
        
        关键：每个Agent只收到其他人的 Cⱼ，不收到 Qⱼ
        Agent必须从 Cⱼ 逆向推断 Agentⱼ 可能拥有什么信息
        """
        attributions = []
        for i, assignment in enumerate(assignments):
            # 构建通信内容：只包含其他人的中间结论
            other_conclusions = [c for j, c in enumerate(conclusions) if j != i]
            
            # Agent被要求做"差异归因"
            attribution = await self._agent_attribute(
                my_conclusion=conclusions[i],
                other_conclusions=other_conclusions,
                prompt_template=DIFFERENCE_ATTRIBUTION_PROMPT
            )
            attributions.append(attribution)
        
        return attributions
    
    async def run_round_2(self, attributions: List[DifferenceAttribution],
                          conclusions: List[IntermediateConclusion]) -> List[IntermediateConclusion]:
        """Round 2: 基于归因修正结论"""
        revised = []
        for attr, original in zip(attributions, conclusions):
            if attr.revision:
                revised_conclusion = await self._agent_revise(original, attr)
            else:
                revised_conclusion = original  # 无需修正
            revised.append(revised_conclusion)
        return revised
```

### 2.3 FusionJudge（融合判断器）

**不是投票器，是矛盾检测器。**

```python
class FusionJudge:
    """融合判断器——检测矛盾，决定回溯或输出"""
    
    # 矛盾分类
    class ContradictionType(Enum):
        ATTRIBUTABLE = "attributable"        # 可归因：从视角差异能解释
        UNATTRIBUTABLE = "unattributable"    # 不可归因：需要回溯
        FALSE_CONSENSUS = "false_consensus"  # 虚假一致：答案同但推理链矛盾
        TRUE_CONVERGENCE = "true_convergence" # 真实收敛：答案+推理链一致
    
    async def judge(self, conclusions: List[IntermediateConclusion]) -> Judgment:
        """
        融合判断核心流程：
        
        1. 提取所有结论的推理链
        2. 逐对比较推理链（不是比较答案！）
        3. 分类矛盾
        4. 决定下一步
        """
        # Step 1: 推理链结构比对
        chain_pairs = self._extract_chain_pairs(conclusions)
        
        # Step 2: 矛盾分类
        contradictions = []
        for pair in chain_pairs:
            ctype = self._classify_contradiction(pair)
            contradictions.append((pair, ctype))
        
        # Step 3: 决策
        judgment = Judgment()
        
        if any(ct == ContradictionType.UNATTRIBUTABLE for _, ct in contradictions):
            # 不可归因矛盾 → 回溯到视角分解器，增补新视角
            judgment.action = "backtrack"
            judgment.reason = "存在无法从视角差异解释的矛盾，需要增补视角"
            
        elif any(ct == ContradictionType.FALSE_CONSENSUS for _, ct in contradictions):
            # 虚假一致 → 最危险，需要深度审查
            judgment.action = "deep_review"
            judgment.reason = "检测到虚假一致：结论相同但推理路径矛盾"
            judgment.review_targets = self._identify_false_consensus_pairs(contradictions)
            
        elif all(ct == ContradictionType.ATTRIBUTABLE for _, ct in contradictions):
            # 可归因矛盾 → 触发Agent间修正通信
            judgment.action = "revision_round"
            judgment.reason = "矛盾可从视角差异解释，触发修正轮次"
            
        else:
            # 真实收敛 → 输出
            judgment.action = "converge"
            judgment.final_answer = self._synthesize_answer(conclusions)
            judgment.reasoning_tree = self._build_reasoning_tree(conclusions)
        
        return judgment
    
    def _classify_contradiction(self, chain_pair) -> ContradictionType:
        """
        矛盾分类的核心逻辑：
        
        关键洞察：不比较答案，比较假设空间
        - Agent₁排除{H₂,H₃}确认H₁（因为证据A否定了H₂/H₃）
        - Agent₂排除{H₁,H₄}确认H₁（因为证据B否定了H₁/H₄）
        → 答案相同(H₁)，但Agent₂的推理链中曾排除H₁！
        → 这是虚假一致的信号
        """
        a_conclusion, b_conclusion = chain_pair
        a_excluded = set(a_conclusion.eliminated_hypotheses)
        b_excluded = set(b_conclusion.eliminated_hypotheses)
        
        # 检查：一方排除的假设是否包含另一方确认的假设
        if a_conclusion.conclusion == b_conclusion.conclusion:
            # 答案相同——但推理链呢？
            if a_excluded & set(b_conclusion.active_hypotheses):
                return ContradictionType.FALSE_CONSENSUS
            elif b_excluded & set(a_conclusion.active_hypotheses):
                return ContradictionType.FALSE_CONSENSUS
            else:
                return ContradictionType.TRUE_CONVERGENC
        
        # 答案不同
        if self._can_explain_from_perspective_diff(chain_pair):
            return ContradictionType.ATTRIBUTABLE
        else:
            return ContradictionType.UNATTRIBUTABLE
```

### 3.1 与 DynamicTopologyRouter 的协作

```python
# 在 dynamic_router.py 中增加协同推理路由
class DynamicTopologyRouter:
    
    async def route_task(self, task: Task) -> RoutePlan:
        """
        路由决策：
        - 简单任务 → 单Agent执行（不需要协同推理）
        - 复杂任务 → 评估是否需要认知分工
        
        判断标准：
        1. 任务涉及多个知识域？→ 是 → 认知分工
        2. 任务需要假设检验？→ 是 → 认知分工
        3. 任务有对抗性论证需求？→ 是 → 角色约束策略
        4. 任务涉及多源证据？→ 是 → 证据拆分策略
        """
        if self._needs_collaborative_reasoning(task):
            plan = self.decomposer.decompose(task.question, self.agents)
            execution_plan = ExecutionPlan(
                mode="cognitive_division",
                perspective_plan=plan,
                max_rounds=2,
                agents=[self._get_agent(a.agent_id) for a in plan.assignments]
            )
        else:
            execution_plan = ExecutionPlan(
                mode="sequential",
                agents=[self._select_best_agent(task)]
            )
        
        return execution_plan
```

## 四、量化指标体系

| 指标 | 定义 | 计算方式 | 意义 |
|------|------|---------|------|
| **协同增益比** | 认知分工 vs 最强单Agent的准确率提升 | CDoL_accuracy / max(single_agent_accuracy) | >1说明有增益 |
| **虚假一致检出率** | 融合判断器成功识别虚假一致的比例 | true_positives / (true_positives + false_negatives) | 核心指标 |
| **共识收敛速度** | 达到真实收敛所需的通信轮数 | 轮次数 | 越少越好 |
| **归因准确率** | Agent差异归因的正确率 | 正确归因数 / 总归因数 | 衡量逆向推断能力 |
| **视角桥接度** | 视角间信息差被成功弥合的比例 | 成功桥接的矛盾数 / 总矛盾数 | 视角分解质量的间接指标 |
| **推理链深度** | 最终推理树的平均深度 | 推理链步骤数 | 认知分工是否产生了更深的推理 |

## 六、演示叙事策略

**演示场景**：低碳水泥配方优化中的关键决策——"偏高岭土 vs 石灰石粉作为SCM的优选方案"

**演示结构**：

1. **展示信息不对称**：
   - Agent₁（Miner）只看到3篇支持偏高岭土的文献
   - Agent₂（Assayer）只看到3篇支持石灰石粉的文献
   - 评委看到两个Agent各自的视角

2. **展示推理过程**：
   - Round 0：Agent₁结论"偏高岭土更优"，Agent₂结论"石灰石粉更优"
   - Round 1：Agent₁看到Agent₂的结论后→"对方可能看到了我没看到的耐久性数据"
   - Agent₁修正："在耐久性要求高时，石灰石粉可能更优"
   - Agent₂修正："在早期强度要求高时，偏高岭土可能更优"

3. **展示融合判断**：
   - Judge检测到：两个Agent的推理链中，排除对方选项的理由不同
   - 不是简单的"各打五十大板"，而是识别出"两种SCM在不同约束条件下各有优势"
   - 最终输出：混合方案 + 适用条件矩阵

4. **对比基线**：
   - 单Agent（拿到全部6篇文献）→ 倾向于文献数量多的那一方
   - 传统辩论（全量信息共享）→ 收敛到某一方的论点
   - 认知分工 → 得出了"条件依赖型混合方案"，这是任何单Agent（即使拿到全部信息）都不容易自发得出的结论
