# 方案二：自适应上下文管理器 (Adaptive Context Manager)

> 直接受清华 & OpenBMB 论文《Rethinking Hybrid Attention》(arXiv:2606.15378) 启发

---

## 论文核心发现回顾

| 发现 | 含义 |
|------|------|
| 全注意力层承载长程信息检索能力 | 真正重要的不是"看到多少"，而是"主动检索多远" |
| 高效注意力只影响涌现速度，不影响上限 | 辅助模块决定学习效率，但最终能力由核心模块决定 |
| **大窗口懒惰症** | 当局部窗口足够大时，模型不再主动学习长程检索——"太舒服就不努力" |
| 小窗口+NoPE有效 | 限制局部可见范围 + 打破位置依赖 → 迫使核心层学习真正的长程能力 |
| 检索头专门化 | 少数注意力头专门负责长程检索，不是所有头都做同样的事 |

---

## 核心类比：从 LLM 到多Agent系统

| LLM 概念 | Agent 系统映射 | 工程含义 |
|----------|---------------|---------|
| 全注意力层 = 长程能力承载者 | **全局记忆池 (Global Memory Pool)** | 所有Agent的推理结论汇聚处，Router维护 |
| 高效注意力 = 局部处理 | **Agent本地上下文 (Local Context)** | 每个Agent自己看到的信息，有限窗口 |
| 滑动窗口大小 | **Agent本地上下文窗口大小** | 窗口越大，Agent越"自给自足"，越不主动求助 |
| 大窗口懒惰症 | **上下文舒适区陷阱** | Agent拥有足够多本地信息时，不再主动通过Router获取远程信息 |
| 检索头 (Retrieval Head) | **检索头Agent (Retrieval Head Agent)** | 专门化的信息桥接Agent，负责跨域长程检索 |
| NoPE（去掉位置编码） | **强制全局同步 (Forced Global Sync)** | 定期打破信息壁垒，迫使Agent重新评估全局状态 |
| 注意力熵（检索精度指标） | **信息检索精度 (Retrieval Precision)** | Agent从全局记忆中精准获取相关信息的能力 |

## 二、核心机制

### 2.1 小窗口策略：限制本地上下文

**论文洞察**：大窗口导致懒惰——当滑动窗口足够大时，模型依赖局部信息就能完成大部分预测，不需要学习长程检索。

**Agent系统映射**：当Agent的本地上下文足够大（能看到所有历史对话、所有检索结果），Agent就不需要主动通过Router获取其他Agent的信息。它变得"自给自足"，协同退化为各自为政。

**工程实现**：

```python
@dataclass
class LocalContextWindow:
    """Agent的本地上下文窗口——故意设小"""
    
    max_tokens: int = 4096          # 硬上限：每个Agent最多看到4K tokens
    strategy: str = "recency"       # 截断策略：recency | relevance | mixed
    
    # 动态调整
    adaptive: bool = True           # 是否自适应调整窗口大小
    min_window: int = 1024          # 最小窗口（强制限制）
    max_window: int = 8192          # 最大窗口（即使任务复杂也不超过此值）
    
    def truncate(self, context: List[Message]) -> List[Message]:
        """
        截断上下文到窗口大小。
        
        关键：截断不是"丢弃"，而是"降级到全局记忆池"
        - 被截断的信息仍然存在于Global Memory Pool
        - Agent需要时可以通过Router检索
        - 但默认看不到——迫使Agent主动检索
        """
        total_tokens = sum(msg.token_count for msg in context)
        
        if total_tokens <= self.max_tokens:
            return context
        
        if self.strategy == "recency":
            # 保留最近的N条消息
            return self._keep_recent(context, self.max_tokens)
        elif self.strategy == "relevance":
            # 保留与当前子任务最相关的消息
            return self._keep_relevant(context, self.max_tokens)
        elif self.strategy == "mixed":
            # 最近3条 + 相关度最高的N条
            recent = context[-3:]
            relevant = self._select_by_relevance(context[:-3], self.max_tokens - sum(m.token_count for m in recent))
            return relevant + recent
```

### 2.2 NoPE策略：强制全局同步

**论文洞察**：对全注意力层应用NoPE（去掉位置编码）能显著提升长上下文性能——因为打破了位置依赖，迫使模型学习真正的语义检索而非位置记忆。

**Agent系统映射**：定期强制所有Agent接收全局状态摘要，打破"信息茧房"。不是Agent主动请求（那又变成"大窗口"了），而是系统强制推送。

```python
class ForcedGlobalSync:
    """
    强制全局同步——NoPE的Agent系统实现
    
    设计意图：
    - 不是Agent主动查询全局状态（那是"大窗口"行为）
    - 是系统定期强制注入全局信息（打破局部依赖）
    - 每次同步后，Agent必须重新评估自己的结论是否仍然成立
    """
    
    def __init__(self, sync_interval: int = 10, summary_model: str = "flash"):
        self.sync_interval = sync_interval      # 每N步触发一次同步
        self.step_counter = 0
        self.sync_count = 0
        self.summary_model = summary_model
    
    async def maybe_sync(self, global_memory: GlobalMemoryPool, 
                         agents: List[Agent]) -> Optional[SyncResult]:
        """
        检查是否需要同步，如果需要则执行。
        
        返回同步结果（注入到每个Agent的下一轮输入中）。
        """
        self.step_counter += 1
        
        if self.step_counter % self.sync_interval != 0:
            return None
        
        self.sync_count += 1
        
        # 1. 生成全局摘要（不是原始数据，是压缩后的摘要）
        global_summary = await self._generate_summary(global_memory)
        
        # 2. 检测自上次同步以来的关键变化
        changes = self._detect_changes(global_memory, self._last_sync_snapshot)
        
        # 3. 构建同步消息
        sync_message = SyncMessage(
            sync_id=self.sync_count,
            global_summary=global_summary,
            key_changes=changes,
            contradictions_detected=self._detect_cross_agent_contradictions(global_memory),
            action_required="re-evaluate your conclusion based on global state"
        )
        
        # 4. 注入到每个Agent（强制，不是请求）
        self._last_sync_snapshot = global_memory.snapshot()
        
        return SyncResult(sync_message=sync_message, target_agents=[a.id for a in agents])
    
    async def _generate_summary(self, memory: GlobalMemoryPool) -> str:
        """
        生成全局摘要。
        
        关键：摘要是"无位置编码"的——不包含"谁先说、谁后说"的时序信息
        只包含"当前状态是什么、关键矛盾是什么、待解决的是什么"
        """
        prompt = f"""
        当前全局推理状态：
        
        已确认的结论：{memory.confirmed_conclusions}
        活跃假设：{memory.active_hypotheses}
        未解决的矛盾：{memory.unresolved_contradictions}
        各Agent当前立场：{memory.agent_stances}
        
        请生成一份简洁的全局状态摘要（200字以内），重点突出：
        1. 当前最关键的分歧点
        2. 最可能被推翻的结论
        3. 下一步最应该探索的方向
        
        注意：不要按时间顺序叙述，按重要性排列。
        """
        
        return await self.llm.call(prompt, model=self.summary_model)
```

### 2.3 检索头Agent：专门化的信息桥接

**论文洞察**：在混合架构中，少数"检索头"(Retrieval Head)专门负责长程信息检索。这些头发展出特殊的注意力模式——低注意力熵、精准的远程定位能力。

**Agent系统映射**：指定专门的Agent作为"检索头"——不做推理，只做信息桥接。它们的存在让整个系统能处理跨域、跨时段的长程信息需求。

```python
class RetrievalHeadAgent:
    """
    检索头Agent——专门化的信息桥接
    
    与普通Agent的区别：
    - 普通Agent：接收信息 → 推理 → 产出结论
    - 检索头Agent：接收查询 → 检索全局记忆 → 返回精准信息片段
    
    不推理，不判断，只检索。
    
    灵感来源：论文中发现的Retrieval Head——
    这些注意力头的特征是低注意力熵（精准的远程定位），
    而不是高注意力熵（广泛的局部关注）。
    """
    
    def __init__(self, agent_id: str, global_memory: GlobalMemoryPool):
        self.agent_id = agent_id
        self.global_memory = global_memory
        self.retrieval_log: List[RetrievalRecord] = []
    
    async def retrieve(self, query: RetrievalQuery) -> RetrievalResult:
        """
        精准检索——模拟Retrieval Head的低熵注意力模式
        
        query包含：
        - 目标信息描述（"我需要关于X的证据"）
        - 信息约束（"时间在2024年之后"、"来自材料科学领域"）
        - 用途说明（"用于验证假设H1"）——帮助判断信息的相关性
        """
        # 1. 语义检索：从全局记忆中找相关内容
        candidates = await self.global_memory.semantic_search(
            query.target_description,
            top_k=20
        )
        
        # 2. 约束过滤：按时间、领域等过滤
        filtered = self._apply_constraints(candidates, query.constraints)
        
        # 3. 精准排序：根据用途评估相关性
        # 这一步是关键——不是简单的相似度排序，而是"这条信息对当前推理有什么用"
        ranked = await self._rank_by_utility(filtered, query.purpose)
        
        # 4. 返回top-K
        result = RetrievalResult(
            items=ranked[:query.top_k],
            total_candidates=len(candidates),
            retrieval_precision=self._compute_precision(ranked[:query.top_k], query),
            query=query
        )
        
        # 记录检索日志（用于后续分析检索头的"注意力模式"）
        self.retrieval_log.append(RetrievalRecord(
            timestamp=time.time(),
            query=query,
            result_summary=result.summarize()
        ))
        
        return result


@dataclass
class RetrievalQuery:
    """检索头Agent的查询"""
    requester_id: str                    # 谁发起的检索请求
    target_description: str              # 目标信息描述
    constraints: Dict[str, Any]          # 约束条件
    purpose: str                         # 用途（帮助判断相关性）
    top_k: int = 5                       # 返回数量


@dataclass
class RetrievalResult:
    """检索结果"""
    items: List[MemoryItem]              # 检索到的信息项
    total_candidates: int                # 候选总数
    retrieval_precision: float           # 检索精度（用于衡量检索头效果）
    query: RetrievalQuery                # 原始查询
```

## 三、核心创新：上下文舒适区检测与干预

### 3.1 懒惰检测器

**论文洞察**：大窗口懒惰症的核心信号是——当局部窗口足够大时，全注意力层的检索头形成更晚、发展更慢。

**Agent系统映射**：当Agent长期不通过Router检索远程信息时，说明它处于"上下文舒适区"——可能正在错过关键信息。

```python
class LazinessDetector:
    """
    上下文舒适区检测器
    
    检测信号：
    1. 检索频率下降：Agent长时间不向Global Memory发起检索请求
    2. 修正率下降：Agent在Global Sync后几乎不修正自己的结论
    3. 置信度膨胀：Agent的置信度持续上升但不被外部信息挑战
    4. 信息熵下降：Agent的推理链中引用的信息源越来越单一
    """
    
    def __init__(self, window_size: int = 20):
        self.window_size = window_size
        self.metrics_history: Dict[str, List[float]] = defaultdict(list)
    
    def check(self, agent_id: str, agent_state: AgentState) -> LazinessAlert:
        """检查单个Agent是否处于舒适区"""
        
        # 指标1：检索频率
        retrieval_rate = self._compute_retrieval_rate(agent_id, self.window_size)
        
        # 指标2：Sync后修正率
        revision_rate = self._compute_revision_rate(agent_id, self.window_size)
        
        # 指标3：置信度变化趋势
        confidence_trend = self._compute_confidence_trend(agent_id, self.window_size)
        
        # 指标4：信息源多样性
        source_diversity = self._compute_source_diversity(agent_id, self.window_size)
        
        # 综合判断
        laziness_score = self._compute_laziness_score(
            retrieval_rate, revision_rate, confidence_trend, source_diversity
        )
        
        alert = LazinessAlert(
            agent_id=agent_id,
            laziness_score=laziness_score,
            is_lazy=laziness_score > 0.7,
            details={
                "retrieval_rate": retrieval_rate,
                "revision_rate": revision_rate,
                "confidence_trend": confidence_trend,
                "source_diversity": source_diversity
            }
        )
        
        return alert
    
    def intervention(self, alert: LazinessAlert) -> InterventionAction:
        """
        对"懒惰"Agent的干预措施
        
        类比论文中的干预：
        - 论文：对全注意力层应用NoPE → 打破位置依赖
        - Agent系统：强制注入"意外信息" → 打破认知舒适区
        """
        if not alert.is_lazy:
            return None
        
        # 干预策略1：强制检索任务
        # 给Agent一个必须从全局记忆中检索才能回答的问题
        forced_query = self._generate_forced_query(alert.agent_id)
        
        # 干预策略2：注入矛盾信息
        # 从全局记忆中找一条与该Agent当前结论矛盾的信息，强制注入
        contradicting_info = self._find_contradicting_info(alert.agent_id)
        
        # 干预策略3：缩小上下文窗口
        # 如果Agent仍然"懒惰"，进一步缩小其本地窗口
        window_reduction = self._reduce_window(alert.agent_id, factor=0.5)
        
        return InterventionAction(
            agent_id=alert.agent_id,
            forced_query=forced_query,
            contradicting_info=contradicting_info,
            new_window_size=window_reduction,
            rationale="Agent处于上下文舒适区，需要打破局部依赖"
        )
```

### 3.2 自适应窗口调整

```python
class AdaptiveWindowController:
    """
    自适应窗口控制器
    
    核心思想：窗口大小不是固定的，而是根据任务阶段动态调整。
    
    类比论文：
    - 论文发现小窗口+NoPE组合最优
    - 但小窗口不适合所有阶段（初始探索阶段需要较大窗口）
    
    策略：
    - 任务初期：较大窗口（8K），允许Agent快速建立局部理解
    - 任务中期：缩小窗口（4K），迫使Agent开始检索远程信息
    - 任务后期：最小窗口（2K）+ 频繁Sync，聚焦最终决策
    """
    
    def __init__(self, initial_window: int = 8192):
        self.initial_window = initial_window
        self.current_window = initial_window
        self.phase = "exploration"  # exploration | refinement | decision
    
    def adjust(self, task_progress: float, laziness_score: float) -> int:
        """
        根据任务进度和懒惰程度调整窗口
        
        task_progress: 0-1，任务完成进度
        laziness_score: 0-1，懒惰检测分数
        """
        if task_progress < 0.3:
            # 探索阶段：保持较大窗口
            self.phase = "exploration"
            target_window = max(4096, self.initial_window)
            
        elif task_progress < 0.7:
            # 精炼阶段：缩小窗口，迫使检索
            self.phase = "refinement"
            target_window = 4096
            
            # 如果懒惰分数高，进一步缩小
            if laziness_score > 0.5:
                target_window = 2048
                
        else:
            # 决策阶段：最小窗口 + 高频Sync
            self.phase = "decision"
            target_window = 2048
            
            if laziness_score > 0.3:
                target_window = 1024
        
        # 平滑过渡（不跳变）
        self.current_window = int(
            0.7 * self.current_window + 0.3 * target_window
        )
        
        return self.current_window
    
    def get_sync_interval(self) -> int:
        """根据阶段调整Sync频率"""
        if self.phase == "exploration":
            return 20  # 每20步同步一次
        elif self.phase == "refinement":
            return 10  # 每10步同步一次
        else:
            return 5   # 每5步同步一次
```

## 四、全局记忆池设计

```python
class GlobalMemoryPool:
    """
    全局记忆池——全注意力层的Agent系统实现
    
    存储所有Agent的推理产物，支持语义检索。
    
    关键设计：
    - 不存储原始对话（那是Agent的本地上下文）
    - 只存储结构化的推理产物（结论、假设、矛盾）
    - 支持按语义检索（模拟Retrieval Head的精准定位）
    """
    
    def __init__(self):
        self.conclusions: List[MemoryItem] = []      # 推理结论
        self.hypotheses: List[MemoryItem] = []       # 假设
        self.contradictions: List[MemoryItem] = []   # 矛盾记录
        self.evidence: List[MemoryItem] = []         # 证据
        self.retrieval_index = None                   # 语义检索索引
    
    def add_conclusion(self, agent_id: str, conclusion: IntermediateConclusion):
        """Agent提交结论到全局记忆"""
        item = MemoryItem(
            content=conclusion.conclusion,
            metadata={
                "agent_id": agent_id,
                "confidence": conclusion.confidence,
                "reasoning_chain": conclusion.reasoning_chain,
                "timestamp": time.time(),
                "type": "conclusion"
            },
            embedding=self._embed(conclusion.conclusion)
        )
        self.conclusions.append(item)
        self._update_index()
    
    async def semantic_search(self, query: str, top_k: int = 10) -> List[MemoryItem]:
        """语义检索——Retrieval Head Agent的核心依赖"""
        query_embedding = self._embed(query)
        
        # 按相似度排序
        scored = []
        for item in self.conclusions + self.hypotheses + self.contradictions + self.evidence:
            score = cosine_similarity(query_embedding, item.embedding)
            scored.append((score, item))
        
        scored.sort(key=lambda x: x[0], reverse=True)
        return [item for _, item in scored[:top_k]]
```

## 五、与认知分工的协同关系

两个方案不是独立的，而是互补的：

| 维度 | 认知分工引擎 (方案一) | 自适应上下文管理 (方案二) |
|------|---------------------|------------------------|
| 解决的问题 | 如何让Agent产生**不同**的推理路径 | 如何让Agent**不懒惰**地保持长程信息检索 |
| 核心机制 | 主动制造信息不对称 | 限制本地窗口 + 强制全局同步 |
| 论文对应 | 无直接对应（用户原创） | 直接对应论文所有发现 |
| 触发时机 | 复杂任务启动时 | 任务全程持续运行 |
| 关系 | 认知分工**产生**信息不对称 | 上下文管理**维持**信息不对称 |

**协同工作流**：
1. PerspectiveDecomposer 分配视角 → 产生初始信息不对称
2. 各Agent在小窗口下独立推理 → 不能"舒服地"看到所有信息
3. Agent通过Retrieval Head Agent检索全局记忆 → 补充缺失信息
4. CommunicationLayer 传递中间结论 → 有损通信
5. ForcedGlobalSync 定期注入全局状态 → NoPE效果
6. LazinessDetector 监测 → 发现懒惰则干预
7. FusionJudge 融合判断 → 输出

## 六、量化验证方案

### 6.1 小窗口 vs 大窗口对比实验

| 实验组 | 窗口大小 | Sync频率 | 预期结果 |
|--------|---------|----------|---------|
| A (大窗口) | 32K tokens | 无Sync | 基线：Agent各自为政，协同增益低 |
| B (小窗口) | 4K tokens | 无Sync | 中等：Agent被迫检索，但缺乏全局视图 |
| C (小窗口+Sync) | 4K tokens | 每10步 | 较好：小窗口+定期全局注入 |
| D (小窗口+Sync+干预) | 4K tokens | 每10步+懒惰干预 | 最优：完整的NoPE策略 |

**验证指标**：
- 任务完成率
- 推理链深度（平均步骤数）
- 信息源多样性（引用的不同来源数量）
- 协同增益比（vs 单Agent）
- 检索精度（Retrieval Head的命中率）

### 6.2 懒惰检测有效性验证

- 注入"懒惰场景"：给Agent过大的上下文窗口
- 检测懒惰检测器能否识别
- 验证干预后是否恢复活跃检索行为

## 七、最小可行实现（MVP）

```
文件：adaptive_context_manager.py（约500行）

核心类：
├── LocalContextWindow       (~60行)   — 本地上下文窗口管理
├── GlobalMemoryPool         (~100行)  — 全局记忆池 + 语义检索
├── ForcedGlobalSync         (~80行)   — 强制全局同步
├── RetrievalHeadAgent       (~100行)  — 检索头Agent
├── LazinessDetector         (~80行)   — 懒惰检测器
├── AdaptiveWindowController (~50行)   — 自适应窗口控制
└── AdaptiveContextManager   (~30行)   — 编排入口

依赖：numpy（向量计算）+ 现有LLM调用能力
不依赖：networkx、gradio
```
